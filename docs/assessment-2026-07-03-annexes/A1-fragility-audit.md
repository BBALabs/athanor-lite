# Condere Fragility Audit — Concrete Findings

Codebase read in full: `src-tauri/src/**` (lib.rs, error.rs, workspaces/mod.rs, hardware/{mod,gpu,telemetry}.rs, models/{mod,recommend}.rs, catalog.json), `src/state/store.ts`, `src/lib/*.ts`, all views/components, `tauri.conf.json`, `Cargo.toml`, `capabilities/default.json`, `package.json`.

---

## CRITICAL

- **Corrupted/torn `state.json` bricks the app permanently — boot dead-end with a Retry button that can never succeed.**
  `src-tauri/src/workspaces/mod.rs:61-67` — `read_state` propagates any `serde_json` parse error. `list()` (`mod.rs:117`) calls it, `list_workspaces` is boot step 4 (`src/state/store.ts:105-107`), and the boot chain's catch (`store.ts:122-131`) sets `boot: "error"` which renders a full-screen fault (`src/components/BootSequence.tsx:61-68`) whose only affordance is `window.location.reload()` (`BootSequence.tsx:64`) — which re-reads the same corrupt file forever. When hit: power loss / crash / disk-full during any `write_state`, or a user/AV tool touching the file. Fix: treat unparseable `state.json` as `AppState::default()` (it's just an active-pointer cache; losing it is harmless), and log the corruption instead of failing boot.

- **All on-disk writes are non-atomic — `fs::write` truncates then writes.**
  `src-tauri/src/workspaces/mod.rs:70` (`write_state`) and `mod.rs:93` (`write_manifest`). Power loss, crash, or ENOSPC between truncate and flush leaves a zero-byte or half-written JSON file. Consequences: torn `state.json` → the critical boot brick above; torn `workspace.json` → the workspace silently vanishes from the list (`mod.rs:105-112` skips unreadable manifests with only a log line — the user sees their workspace gone with no explanation). Note `activate()` (`mod.rs:171-172`) rewrites the manifest on *every workspace switch*, so the exposure window recurs constantly, not just at creation. Fix: write to `workspace.json.tmp`, `File::sync_all()`, then `fs::rename` over the original (atomic on NTFS/POSIX); on read failure, fall back to a `.bak`/previous copy and surface "workspace X is damaged" in the UI rather than hiding it.

- **Zero disk space turns into data corruption, not an error.**
  Same lines (`mod.rs:70,93`): with 0 bytes free, `fs::write` truncates the existing file, then fails the write → existing good `state.json`/`workspace.json` is destroyed. Given this app targets people downloading 20–50 GB GGUF files, disk-full is a *routine* customer state, not an edge case. The atomic-rename fix above also fixes this (rename never destroys the original on failure).

- **No code signing and no updater — the product can't reach or stay on customer machines.**
  `src-tauri/tauri.conf.json:29-39` — bundle config has no Windows signing (`certificateThumbprint`/signing command), no macOS signing/notarization, and no `plugins.updater` / `createUpdaterArtifacts`; `Cargo.toml` has no `tauri-plugin-updater`. When hit: **first run, every customer** — Windows SmartScreen shows "Windows protected your PC" (unsigned, zero reputation) and most non-technical buyers stop there; macOS Gatekeeper on current macOS refuses unsigned apps outright. And with no updater, every fix below (including the catalog problem) requires customers to manually find and reinstall a new build. Fix: EV or standard code-signing cert + signtool in the bundle step, Apple Developer ID + notarization, and `tauri-plugin-updater` with signed update artifacts before any external release.

## HIGH

- **No schema version and no `#[serde(default)]` on any persisted struct — the first additive field change strands every existing user's data.**
  `src-tauri/src/workspaces/mod.rs:18-39` — `Workspace` and `AppState` have no `version`/`schemaVersion` field and no serde defaults. The roadmap explicitly plans M2+ fields (`model_refs` comment, `mod.rs:31`). The failure is bidirectional: a v2 app adding a required field can't parse v1 manifests (workspace silently skipped → "my workspaces disappeared after the update"); a v1 app reading a v2 manifest (rollback, shared/synced AppData) also fails. Fix now, while there are zero users: add `"schema": 1` to both files, `#[serde(default)]` on every non-essential field, and an explicit migrate-on-read path.

- **No single-instance guard — two app instances race on the same files with a lock that only exists in one process.**
  `src-tauri/src/lib.rs:83` — `WsLock` is an in-process `Mutex<()>`; there is no `tauri-plugin-single-instance` in `Cargo.toml` and no file locking. When hit: user double-clicks the taskbar icon twice, or launches from installer's "Run" while an old instance is open — both processes do unguarded read-modify-write of `state.json` (`mod.rs:158-160`, `174-176`), so last-writer-wins clobbers, and interleaved `fs::write` from two processes can interleave torn output. Also spawns a second telemetry thread and a second NVML session. Fix: add `tauri-plugin-single-instance` (focus existing window on second launch); if multi-instance is ever desired, an OS-level file lock on the data root.

- **Boot is a fail-fast serial chain: any single step failure dead-ends the entire app.**
  `src/state/store.ts:89-131` — one `try/catch` wraps hardware probe → catalog → recommendations → workspaces → telemetry; the first rejection aborts everything and `App.tsx:98` renders nothing unless `boot === "ready"`. When hit: WMI service broken (common on unhealthy Windows installs — `gpu.rs:287-290` handles the *query* failing, but `detect()` itself can still error), or AppData permission problems on `list_workspaces` → customer gets a full-screen "startup fault" and can't even see the model catalog, which needs no hardware at all. Fix: make each step independently degradable — hardware failure should yield a "detection unavailable" dashboard, workspace failure should not block model browsing; reserve the fatal screen for genuinely fatal states.

- **Memory-floor math is only valid at exactly 8K context — recommendations overpromise by 2–10× VRAM for the long-context work the catalog advertises.**
  `src-tauri/src/models/mod.rs:27` (`min_mem_gb` = "weights + KV at 8K + overhead") vs. `catalog.json` entries advertising `contextLength: 131072` shown in the UI (`src/views/Models.tsx:71`). KV cache scales linearly with context: a 70B Q4 at 8K KV fits the 45.5 GB floor, but at 64K context KV alone adds tens of GB — the customer loads the "recommended" model, raises context, and gets an OOM or silent RAM-spill slowdown, on the flagship pick. `recommend.rs:83-99` only mentions "keep context at 8K" in the *tight* branch. MoE entries (gpt-oss-20b) also have no active-vs-total param modeling. Fix: make context a parameter of the fit math (`kv_gb_per_1k_tokens` per entry in the catalog) and show fit at the user's intended context, or clamp the advertised context to what the headroom actually supports.

- **Multi-GPU machines are budgeted as single-GPU, and *used* VRAM is ignored.**
  `src-tauri/src/hardware/mod.rs:104-111` (`max_gpu_vram_gb` — largest single device, acknowledged as M2) and `recommend.rs:61-71` — budget = `total * 0.95 - 0.5`, never consulting `vram_used_bytes`. When hit: (a) a 2×24 GB rig is told its ceiling is a 32B model — half its capability (Tony's own dual-P40 server would be mis-scoped); (b) a card already holding 8 GB (game, another model server, Chrome hardware accel, Windows DWM which routinely takes 1–2 GB on the display GPU) gets a recommendation that instantly OOMs. 0.95 usable is optimistic for the display-attached GPU even when idle. Fix: budget from `total - used` at detection time (data is already collected, `gpu.rs:117-118`), warn when used > ~10%, and sum eligible devices with a tensor-parallel caveat for multi-GPU.

- **macOS / Apple Silicon / Linux+AMD are silently classified as CPU-only garbage-tier.**
  GPU detection is NVML (`gpu.rs:77-132`) plus a Windows-only WMI/registry probe (`gpu.rs:188+`, `Cargo.toml:30-32`); `tauri.conf.json:31` sets `targets: "all"` and the bundle config ships icns. A 128 GB M-series Mac — the single best consumer local-LLM machine that exists — reports `gpus: []` → `ComputeClass::CpuOnly` → budget = 50% RAM with the note "expect a few tokens/sec" (`recommend.rs:85-87`), which is flatly wrong under Metal. Linux+AMD (ROCm) likewise invisible. Also `decorations: false` + custom Windows-style min/max/close (`Titlebar.tsx:49-68`) is wrong chrome on macOS. Fix: either declare Windows/NVIDIA-only for v1 (change bundle targets, say so on the site), or add Metal (`metal` crate / `system_profiler`) and unified-memory-aware budgeting before shipping cross-platform.

- **The model catalog is frozen into the binary with no update path.**
  `src-tauri/src/models/mod.rs:55` (`include_str!("catalog.json")`, version "2026.07"). The local-model landscape turns over monthly; with no updater (see Critical) and no remote catalog fetch, every install's recommendations rot immediately — in 6 months the app confidently recommends models nobody should run, which for a product whose entire value is "we recommend the right model" is an expiration date on the product itself. Fix: keep the embedded catalog as the offline seed, add a signed remote catalog with an on-disk cached copy + `version` comparison; requires the updater work anyway for the fit-math fields above.

## MEDIUM

- **NVML handle is cached once per process — a GPU driver update (or NVML-unavailable moment at launch) kills GPU awareness until app restart.**
  `src-tauri/src/hardware/gpu.rs:26-35` — `OnceLock<Option<Nvml>>`: if init fails at boot (driver mid-update, or the ~30 s after a driver crash/TDR reset), `None` is cached forever → machine mis-classified CpuOnly for the whole session. If init succeeded and the driver restarts mid-session, every `device_by_index` fails → `sample_gpus()` (`telemetry.rs:36-61`) returns `[]` → Dashboard's ring silently flips from "VRAM" to "memory" and the GPU/VRAM band cells vanish (`Dashboard.tsx:59-64, 178-195`) with zero explanation, and never recover. When hit: GeForce Experience/NVIDIA App auto-updates drivers while Condere is open — common. Fix: on N consecutive NVML failures, drop and re-init the handle; surface "GPU telemetry lost — driver restarted?" in the UI.

- **Telemetry sampler has no staleness contract: one emit failure permanently stops it, and the UI presents the last sample as live forever.**
  `src-tauri/src/hardware/telemetry.rs:86-90` — *any* `app.emit` error breaks the loop permanently (assumed to mean shutdown). Frontend: `useLatestSample` (`store.ts:169-171`) returns the last sample with no timestamp check; `StatusBar` (`App.tsx:43-44`) shows "telemetry 1 Hz" green as long as *any* sample was ever received. After system sleep/resume there's also a visible gap plus a possible stall. When hit: sleep/resume, webview hiccup, transient emit failure → dashboard silently freezes at stale numbers, indistinguishable from a healthy idle machine. Fix: only break on shutdown-class errors (retry with backoff otherwise); frontend should compare `tsMs` to `Date.now()` and grey out readouts older than ~5 s.

- **Poisoned-mutex cascade: five `.unwrap()`s on the workspace lock.**
  `src-tauri/src/lib.rs:32, 45, 55, 65` — `lock.0.lock().unwrap()`. If any workspace op ever panics while holding the guard, the mutex poisons and **every subsequent workspace command panics** for the session — one bad op bricks the whole workspace feature. Also `lib.rs:89` `.expect("telemetry thread must spawn")` panics the app at startup on thread-spawn failure. Fix: `lock().unwrap_or_else(|p| p.into_inner())` (the lock guards no invariant-carrying data — it's a `Mutex<()>`), and degrade telemetry to "unavailable" instead of expecting.

- **Workspace delete is instant, permanent, recursive `remove_dir_all` behind a 3.5-second double-click.**
  `src-tauri/src/workspaces/mod.rs:188` + `src/views/Workspaces.tsx:16-40` (armed state auto-resets at 3.5 s; second click deletes). Today a workspace is one JSON file; the M2+ plan puts documents/models/memory in that directory — at which point two clicks 3 s apart permanently destroy a customer's document set. No trash, no undo. Also: on Windows, `remove_dir_all` fails midway if any file is locked (search indexer, AV, the model file being read) → manifest may already be gone → workspace vanishes from the UI leaving orphaned gigabytes on disk invisibly. Fix: move-to-trash (`trash` crate) or rename to `.deleted-{ts}` with delayed purge; delete manifest *last* so partial failures stay visible.

- **Design harness ships inside the production bundle and is one falsy global away from showing every customer fake hardware.**
  `src/lib/ipc.ts:49` — `export const ipc: Ipc = IN_TAURI ? tauriIpc : (harnessIpc as Ipc)` with `IN_TAURI = "__TAURI_INTERNALS__" in window` (`tauriEnv.ts:2-3`), and `designHarness.ts` is statically imported so the entire synthetic Threadripper/RTX-6000 profile, fake workspaces, and fake telemetry are in the shipped JS. If the internals global is ever absent (Tauri internals rename, webview init race, someone opens `dist/` in a browser), the app runs convincingly on fabricated data — the only tell is a small titlebar badge (`Titlebar.tsx:42-46`). Also creates build coupling: `designHarness.ts:10` imports `../../src-tauri/src/models/catalog.json`, so the frontend build reaches into the Rust tree (breaks if dirs are ever split), and the fit math is hand-mirrored (`designHarness.ts:67-102`) — the mirror's `note` strings have already drifted from `recommend.rs` (no CPU/tight branches). Fix: `import.meta.env.DEV`-gate the harness behind a dynamic import so tree-shaking removes it from production; move `catalog.json` to a shared top-level location both sides reference.

- **`fitVerdict` labels models up to 15% *over* budget as "tight" — i.e., "fits-ish" for models that do not fit.**
  `src/views/Models.tsx:24-28` — `minMemGb <= budgetGb * 1.15 → "tight"`. On GPU inference, 15% over VRAM budget means RAM spill (5–10× slowdown) or OOM depending on runtime. A customer picks a "tight" quant and gets an experience that contradicts the app's core promise of honest fit verdicts. Fix: "tight" should live *inside* the budget (e.g., 85–100%), over-budget is "exceeds", possibly with a separate "partial offload" verdict that says what it means.

- **Boot-critical `list_workspaces` also *writes* during read: self-heal rewrites `state.json` inside `list()`.**
  `src-tauri/src/workspaces/mod.rs:119-124` — healing a dangling active id calls `write_state` during what the caller thinks is a read. On a read-only data dir, roaming-profile sync conflict, or disk-full, listing workspaces *fails because a write failed*, feeding the boot dead-end. Fix: heal in memory, persist opportunistically, never fail the list on the heal-write.

- **`activateWorkspace`'s error path can throw out of its own catch.**
  `src/state/store.ts:152` — `set({ workspaces: await ipc.listWorkspaces(), ... })` inside `catch`: if the reconciling `listWorkspaces` also rejects (data dir gone, the boot-brick scenario mid-session), the rejection escapes as an unhandled promise rejection and the optimistic UI state (line 147) is left pointing at a workspace that isn't active. Fix: nested try/catch; keep prior state on double failure.

- **First-run trust + distribution polish gaps compound: no publisher metadata beyond `com.bba.condere`, `targets: "all"` produces unsigned artifacts for three OSes from a Windows-only-tested probe stack.**
  `tauri.conf.json:5,31`. A macOS or Linux artifact produced by this config would ship the CpuOnly misclassification (above) to platforms it's never been run on. Fix: restrict targets to what's actually tested (`nsis`/`msi`), revisit per-platform at each port.

## LOW

- **`sysinfo = "=0.33.1"` exact pin** (`Cargo.toml:23`): freezes bug/security fixes and new-hardware support (CPU brand strings, hybrid-core counts on future chips) with no compatible-patch drift; combined with no updater, a sysinfo misdetection on next-gen hardware is permanent until a manual reinstall. Fine pre-release; revisit before GA and document *why* it's pinned.
- **Log files are capped at tauri-plugin-log defaults (~40 KB, keep-one rotation)** (`lib.rs:72-81` — no `max_file_size`/`rotation_strategy` configured): not unbounded growth (good), but the opposite failure — by the time a customer reports a bug, the boot-time hardware detection log that would diagnose it has rotated away, and frontend `console.error` output (`store.ts:123`) is never persisted at all. Fix: `RotationStrategy::KeepAll` with a count cap (e.g., 5 × 1 MB) + attach a webview log target; this is the app's only diagnostics channel — there is no crash reporting of any kind.
- **`workspace_dir` id validation admits Windows reserved device names** (`workspaces/mod.rs:76`): "CON", "NUL", "COM1" pass the alnum-and-dash check; a hand-crafted IPC call would create paths with reserved names causing undefined filesystem behavior. Theoretical (ids are app-generated UUIDs), but the check exists to be defensive — add a UUID-shape check.
- **200 workspaces: functional but O(n) full re-read on every create/activate/delete**, ×2 because the store re-fetches after each op (`store.ts:137,150,152`), and every activation rewrites that workspace's manifest (`mod.rs:171-172`) — 200 workspaces ≈ 400 file reads + 2 writes per switch. Works today; couple it with the non-atomic-write finding and frequent switching maximizes corruption exposure. The shelf UI (`Workspaces.tsx:163-186`) renders all tiles unvirtualized — acceptable at 200, revisit at 1000.
- **`minWidth: 1100`** (`tauri.conf.json:18`) exceeds the effective 1092 CSS px of a 1366×768 laptop at 125% scaling — the window won't fit the single most common budget-laptop display; those are exactly the CpuOnly customers the recommender still serves.
- **Boot "Retry" is a full page reload** (`BootSequence.tsx:64`): works only because `initStarted` (`store.ts:57`) is module state that a reload resets — fragile coupling worth a comment or an explicit re-init action.
- **`get_recommendations` round-trips the `HardwareReport` through the frontend** (`lib.rs:21`, `store.ts:101`): recommendations can be computed from stale/tampered frontend state; harmless today, but recompute server-side from the last detection once anything security- or billing-relevant hangs off it.

---

**Top 5 to fix before any customer touches it:** (1) atomic writes + tolerate corrupt `state.json`, (2) schema version + serde defaults on persisted formats, (3) code signing + updater, (4) single-instance guard, (5) context-aware memory-floor math (or an honest 8K-only label). Items 1, 2, and 4 are hours of work each and eliminate the entire "my workspaces disappeared / app won't start" class.
